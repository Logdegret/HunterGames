How to write a README people can actually use

A good README should answer the questions someone has before they clone, play, test, or contribute. It does not need to be huge. It needs to be honest, specific, and easy to scan.

1

Say what it is

Open with one plain sentence: what the project does, who it is for, and why it exists. Avoid vague lines like "a cool app" or "my project."

2

Show how to run it

Include the exact command or link someone needs first. If there are extra steps, list them in order and mention anything that is optional.

3

Explain the status

Tell visitors what works, what is unfinished, and what you want to improve next. A clear limitation is better than pretending everything is polished.

4

Add useful proof

Screenshots, controls, demo links, credits, license notes, and a small changelog make the project easier to trust and easier to return to later.

Summer 2026

Fun things to build or follow right now

These are current picks for project energy: places to build, write, and share something instead of just scrolling. I will keep this section small so it stays curated.

Stardance by Hack Club

Stardance is a free online summer program for teens 13-18 where you make technical projects, log hours, and earn prizes. It runs June 1 through September 30, 2026, so it fits the build-something-this-summer mood perfectly.

Visit stardance.space

Make a tiny launch page

Pick one project and give it a page with a title, screenshot, what changed this week, and a link to try it. A small clear page is better than a big unfinished redesign.

Keep a shipping log

Once a week, write three bullets: what shipped, what broke, and what you learned. Those notes become README updates, portfolio content, and better commit messages.
